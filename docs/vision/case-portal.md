# 案件ポータル構想（北極星）

最終更新：2026-06-20

report-app-justdb が向かう先（WHY / WHERE）を描く**北極星**。報告書（写真報告書／防除作業報告書／融合版）を1サイトに統合し、最終的に**案件単位の顧客ポータル**へ育てる構想。

## 0. 位置づけ — これは「構想」であって「現況」でも「計画」でもない

寿命の違う3種類の文書を**混ぜると腐る**ので分離する。本書は滅多に変わらない構想。

| 種類 | 問い | 変わる頻度 | 文書 |
|---|---|---|---|
| **構想（北極星）** | WHY / WHERE | 滅多に変わらない | **本書** |
| **現況アーキ** | WHAT IS（今あるもの） | 実装が出たら更新 | `architecture/overview.md`（防除WEBアプリ部分）／`architecture/slack-photo-report-architecture.md`（写真報告書・本番稼働） |
| **計画（リング）** | HOW / WHEN | 頻繁に変わる | 本書 §9（粗いリングのみ）→ 各リング詳細は着手時に `docs/spec/` へ |

横断方針：root [`decisions.md`](../../../decisions.md)（**D-PORTAL** ＝本構想／D-AIDATA／D-DIGEST／D-PORTS）。

## 1. 背骨 — 案件を中心に成果物がぶら下がる

すべては**案件**を背骨に組み立てる。案件の identity・所在は既に正本がある。

```
案件（JUST.DB＝正本：案件ID・GoogleDrive URL・Slack thread_ts/channel）
  ├─ Google Drive 案件フォルダ（写真・資料・_ai/）
  ├─ Slack 案件スレッド（1案件1スレッド）
  └─ digest.md（案件知識の AI 正本要約・D-DIGEST / Phase D2）
```

案件に対し、段階ごとの成果物（報告書・見積・請求）がぶら下がる。**digest.md が連結組織**で、どの成果物も同じ案件文脈を共有する（先行実装した案件ダイジェストが、ポータル拡張の土台になる）。

## 2. 段階モデル — 報告書は案件ライフサイクルに沿って登場する

| 段階 | きっかけ | 主な成果物 | ビジネス上の効き目 |
|---|---|---|---|
| **調査・提案** | 調査 → 見積作成 | 調査写真報告書 ＋ 調査見積書（提案を写真で説明） | **成約率** |
| **受注・施工** | 受注 → 施工 | 施工写真報告書（初回・重要：見積で約束したことの履行を示す） | **追加引き合い**・売上/請求に直結 |
| **点検・継続** | その後の点検作業 | 施工写真報告書2（品質維持）。労力がかかり割愛されがち → 割愛時は **防除作業報告書／融合版** で代替 | 品質の継続証明 |
| **請求** | 施工/点検後 | 請求書（**正本は会計/JUST.DB、ポータルは提示ミラー**） | 透明性・顧客満足 |

顧客から見た価値＝**1案件1URL**で「何を提案され（見積＋調査写真報告）／何が施工され（施工写真報告）／その後どう維持され（点検報告）／いくら請求されたか（請求書）」が**すべて見える**。

## 3. 書類の分類

| 書類 | 性格 | 重さ・リスク |
|---|---|---|
| **写真報告書** | 写真でしっかり説明（調査用／施工用） | 軽（`/report/photo` 稼働） |
| **防除作業報告書** | テキスト中心 | 中（防除modeの実体化が要・A資産流用） |
| **融合版** | 写真報告書ほど作り込まず、テキストだけの防除報告でもない**中量級** | 中（要定義・§未確定論点） |
| **見積書** | 提案の金額・内訳 | 中（JUST.DBに受注ID/見積あり＝“提示”は軽い／“作成”は中） |
| **請求書** | 請求金額 | **重・要慎重**（税・会計・法。**作らず提示ミラーのみ**） |

## 4. 2つの画面 — 見る人で分ける

- **社内＝管理画面**：作成・AI生成・赤丸注記・版管理（現 `/report/photo` の系統）。IAP（@seibu-s.co.jp 限定）。
- **顧客＝提示画面**：URL共有で見せる（郵送/メールの手間を減らす）。**IAP 外**＝認可モデルが社内とは別になる（§8）。

## 5. 提示から「やり取り」へ — 双方向化の軸

ポータルの発展は **一方向「見せる」→ 双方向「やり取りする」** の一本の軸上にある。発展機能（§A）は全てこの層に整理できる。

| 層 | 向き | 例 |
|---|---|---|
| **見せる**（土台） | 社→顧客 | 報告書・見積・請求の提示 |
| **届ける** | 社→顧客（outbound） | 通知（見積届いた等）／いつでも解約／クロスセル広告 |
| **受け取る** | 顧客→社（inbound） | 発注**意向**・追加注文**意向**・質問 |
| **繋ぐ** | 双方向 | 希望施工日 × 自社スケジュールのマッチング（既存 visit-planner に接続） |
| **聞く** | 顧客→社（別チャネル） | フィードバック／アンケート（**案件URLとは別URL**） |

## 6. 設計の2原則 — 「無限の発展性」を軽量に保つ歯止め

- **原則A：意向どまり（intent, not transaction）。** 発注も追加注文も「**発注動機を社に伝える**」まで。**契約成立・課金・入金消込はやらない**。これが請求書の重さ（税・会計・法）を回避し、サイトを軽量に保つ。請求書を「提示ミラーのみ」とするのと同根。
- **原則B：目的でURLを分ける。** 案件提示URL／アンケートURL／（将来）通知・解約設定 は別レーン。機密度・認可・体験を**目的ごとに分離**する。

## 7. 既存資産マップ（北極星 ↔ 現況）

| 構想上の役割 | 既存資産 | 状態 |
|---|---|---|
| 写真報告書・社内作成面 | `/report/photo`（worker＋Slack＋digest） | **本番稼働**（設定に「種類＝調査/施工」あり＝作り分け半分済） |
| 防除作業報告書 | `/mock`・`/report/new`（紺谷V・薬剤カスケード） | **モックで凍結**（要・社内作成画面として実体化） |
| 融合版 | — | 実体なし（§未確定論点で定義してから新規） |
| 顧客提示画面 | — | 完全新規 |
| 希望日マッチング | `seibu-scheduling-platform`（visit-planner） | 既存＝**接続する、作り直さない** |
| 通知 | `justdb-hub-gas` | 既存ハブに乗せる |
| 案件文脈 | digest.md（D-DIGEST/D2） | 稼働＝ポータルの連結組織 |

## 8. 横断制約（ブロッカーではなく設計前提）

1. **顧客提示＝外部公開の認可**：社内（IAP）と顧客（IAP外）は別モデル。提示は「署名付き・期限付き・読み取り専用URL」から。双方向（書き込みを外部から受ける）は**なりすまし・スパム・監査**が増え、一段重い本人性が要る。
2. **JUST.DB API 5000回/日**：見積/請求のライブ取得は予算内に収める（現状の最優先制約）。
3. **同意の世界**：通知・広告は特定商取引法・個人情報保護・オプトアウト（「いつでも解約可能」）を**最初から**設計に入れる（後付けは負債）。
4. **D-AIDATA**：顧客データは学習に使わない閉空間（Anthropic Team/API）にのみ渡す。お金/PIIが乗る層ほど厳守。

## 9. リング計画（粗）— 同心円で外へ

北極星は**ビルドキューではない**。壮大さに引っ張られて最初の一歩を変えない。

- **リング0（done）**：写真報告書・社内面（`/report/photo`）。
- **リング1**：防除作業報告書 mode の実体化 ＋ 融合版の定義 ＋ **顧客提示ビュー**（読み取り専用・署名URL）を写真報告書で先行。
- **リング2**：見積書（提示 → 作成）。
- **リング3**：請求書（**提示ミラーのみ**）。
- **リング4+**：やり取り層（§5）。

各リングの詳細実装計画は着手時に `docs/spec/` へ起こす（構想と計画を分離＝構想が膨らんでも計画は荒れない）。

## A. developability（付録・北極星の射程）

双方向層（§5）に整理される将来機能。射程の記録であって着手約束ではない。

- 通知（見積届いた等）／いつでも解約（オプトアウト）
- 発注意向・追加注文意向（原則A＝課金しない）
- 質問の受付
- 希望施工日 × 自社スケジュールのマッチング（visit-planner 接続）
- フィードバック／アンケート（原則B＝別URL・口頭で言いにくいことも拾う）
- クロスセル広告（自社サービス：例「ネズミ解決後にシロアリ点検」）
