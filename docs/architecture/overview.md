# アーキテクチャ概要（v1.0）

## 1. 背景と目的

西武消毒の**東京支店・城東支店**で乱立する報告書業務を、共通の仕組みに寄せるDX。
発端は、ある顧客の監査で月替りに異なる様式が届きクレームになったこと。
本丸は**施工報告書（防除作業報告書）**——最も現場に刺さるため。

戦略は「**統合の手前**」: いきなり統一を強制すると各人が自バージョンを主張して進まない。
WEBでタブ切替により各様式（紺谷V／写真報告書／融合）を安く・並べて出せるようにし、
日常的に見比べる「まな板」を作って収束を促す。

## 2. 役割分担

| 層 | 役割 |
|---|---|
| **JUST.DB** | 正本（システム・オブ・レコード）。事務データに特化。 |
| **Supabase** | 取得を軽くする層。薬剤資材のミラー＋ケースデータ。 |
| **WEBアプリ** | 作成・AI/テンプレ・出力（PDF）。現場処理を外だし。 |
| **Google Drive** | 成果物の保管（PDF＋再編集JSON＋写真）。 |

JUST.DBのGUIへのこだわりがDXの障害になっていたため、現場処理はWEBへ外だしする。

## 3. データの流れ

```
JUST.DB（正本）
  ├─ 薬剤資材テーブル ──(同期)──▶ Supabase（ミラー）──┐
  └─ 施工日程/案件 ───────────────▶ Supabase ────────┤
                                                      ▼
施工予定ID ──▶ WEBアプリ（/mock）でケース取得＋カスケード入力＋報告書作成
                                                      │
                        ┌─────────────────────────────┤
                        ▼                             ▼
              Google Drive（PDF＋JSON、管理番号で連結）   JUST.DBへ書き戻し
                                                       （金額/回数/薬剤/要約のみ）
```

- **アンカー = 施工予定ID**（報告書↔施工が1:1）。`施工予定ID → 案件ID → 受注ID/見積書`。
- 詳細は `docs/architecture/justdb-supabase-integration.md`。

## 4. 保存（Google Drive）

- 保存対象（既定=方式B: 帳票に割り当てた写真のみ）:
  - `report-{管理番号}.pdf`（清書）
  - `report-{管理番号}.json`（再編集用）
  - `photo-*.jpg`（帳票使用分のみ、圧縮後）
- **Drive＝正本ドキュメント、JUST.DB＝そこから派生した書き込み専用の射影**（二重保管のズレ防止）。
- PDF生成は明示操作時のみ。不要な再出力を避ける。
- 保存失敗時はAPIで失敗を返しリトライ可能に。部分成功は状態を記録し再実行可能に。

## 5. JUST.DBへの書き戻し（方針転換）

旧方針「書き戻さない」は**見直し済み**。報告書はテキスト全文ではなく、
**限定した構造化フィールド（金額・回数・薬剤）＋AI要約**のみをAPIで書き戻す。
写真は重い/高いためJUST.DBに入れず、Driveに置く。

## 6. AI / テンプレ

- 駆除作業報告の自由文は、施工内容テーブル＋定型文から生成（まずテンプレで足り、将来AI）。
- 報告内容は調査報告書・見積書から導出可能（やるべき作業＝報告項目）。本格活用は将来段階。

## 7. 実行基盤

- 候補はVercel（無料枠基準）。モックはローカル＋Supabaseで稼働。
- 注意: ローカルFS保存はVercelでは永続しない。永続はDrive（＋Supabase）が担う。
- デプロイ手順は `docs/runbook/deploy-vercel.md`。

## 8. 段階（フェーズ）

1. モック（紺谷V/写真/融合のタブ切替）＋カスケード＋ケース取得 ← **現在地: 達成**
2. 保存→Drive（PDF＋JSON）、赤丸注記
3. JUST.DB実同期（薬剤資材取得→upsert）＋書き戻し、見積/AI下書き
4. 融合様式の作り込み、東京支店展開、フォーマット標準化
