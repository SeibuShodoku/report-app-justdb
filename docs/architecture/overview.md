# アーキテクチャ概要（v1.0）

> [`vision/case-portal.md`（案件ポータル構想）](../vision/case-portal.md)、写真報告書の現況は
> [`architecture/slack-photo-report-architecture.md`](./slack-photo-report-architecture.md)。

## 1. 背景と目的

西武消毒の**東京支店・城東支店**で乱立する報告書業務を、共通の仕組みに寄せるDX。
発端は、ある顧客の監査で月替りに異なる様式が届きクレームになったこと。
本丸は**施工報告書（防除作業報告書）**——最も現場に刺さるため。

戦略は「**統合の手前**」: いきなり統一を強制すると各人が自バージョンを主張して進まない。
WEBでタブ切替により各様式（紺谷V／写真報告書／融合）を安く・並べて出せるようにし、
日常的に見比べる「まな板」を作って収束を促す。

## 2. 役割分担

| 層 | 役割 |
|---|---|
| **JUST.DB** | 正本（システム・オブ・レコード）。事務データに特化。API上限 5000回/日。 |
| **Supabase** | マスタ（薬剤資材・顧客）のミラー。ケースはミラーせず起動時にライブ取得。 |
| **WEBアプリ** | 作成・AI/テンプレ・出力（PDF）。現場処理を外だし。 |
| **Google Drive** | 成果物の保管（PDF＋再編集JSON＋写真）。 |

JUST.DBのGUIへのこだわりがDXの障害になっていたため、現場処理はWEBへ外だしする。
連携は「収束・冪等・一方向」「データプレーン/コントロールプレーン分離」「契約は通常運転だけを縛る」を
原則とし、最優先制約は **API 5000回/日** ―― 詳細と現状の予算調査は
`docs/architecture/justdb-supabase-integration.md`。

## 3. データの流れ

```
JUST.DB（正本／5000回/日）
  ├─ 薬剤資材・顧客マスタ ──(Lane A: 更新日時で増分pull/日次)──▶ Supabase（マスタ・ミラー）
  │                                                                      │
  └─ 施工日程/案件/見積 ──(Lane B: 起動時に施工予定IDで1件ライブ取得・read-only)──┐  │
                                                                          ▼  ▼
                          施工予定ID ──▶ 報告書サイト：ケース取得（読むだけ）＋カスケード入力＋報告書作成
                                                                          │
                                                                          ▼
                                Google Drive（PDF＋JSON＝正本、管理番号で連結）
                                                                          │
                                                                          ▼
                                確定マニフェスト（case_deliverables）＝案件×時系列の「陳列」
                                ＝社内面と顧客面を 同一JSON・同一レンダラ で描く（同形）

  ※ JUST.DB への書き戻しは報告書サイトの責務ではない。将来の「Schedule／終了報告」面に属する（§5）。
```

- **アンカー = 施工予定ID**（報告書↔施工が1:1）。`施工予定ID → 案件ID → 受注ID/見積書`。
- 報告書サイトの仕事は **陳列（JSON マニフェスト）**。JUST.DB へは **読むだけ（Lane B）／書き戻さない**。
- レーン設計（マスタ=ミラー / ケース=ライブ取得 / 書き戻し=将来の終了報告）と API予算・原則の詳細は
  `docs/architecture/justdb-supabase-integration.md`。

## 4. 保存（Google Drive）

- 保存対象（既定=方式B: 帳票に割り当てた写真のみ）:
  - `report-{管理番号}.pdf`（清書）
  - `report-{管理番号}.json`（再編集用）
  - `photo-*.jpg`（帳票使用分のみ、圧縮後）
- **Drive＝正本ドキュメント**。案件×時系列の見せ方は **確定マニフェスト（JSON）** が担い、社内面と顧客面を同形に描く（§5）。
- JUST.DB への射影（書き戻し）は報告書サイトではなく、将来の **Schedule／終了報告** が担う（§5）。
- PDF生成は明示操作時のみ。不要な再出力を避ける。
- 保存失敗時はAPIで失敗を返しリトライ可能に。部分成功は状態を記録し再実行可能に。

## 5. JUST.DBへの書き戻しは「報告書サイトの仕事ではない」（論点の切り分け・2026-06-21）

報告書サイトの仕事は **案件×時系列の「陳列」**（確定マニフェスト＝JSONで社内面と顧客面を同形に描く・§4）。
JUST.DB へは **読むだけ（ケース文脈＝Lane B）／書き戻さない**。

「金額/回数/薬剤/要約 を JUST.DB に書き戻す」という旧論点（旧「書き戻さない→限定フィールドは戻す」の方針転換）は、
本来 **施工予定（Schedule）の「終了報告」** に属するもので、報告書サイトに混ざっていた。終了報告は同サイトの
**別サーフェス・別ディレクトリ（将来実装）** で、そこが JUST.DB 書き戻し（＝Lane C）の正しい置き場。
詳細は `architecture/justdb-supabase-integration.md` Lane C。

**帰結**：報告書サイトは JUST.DB に書かない＝読みだけ＝低コール。**API 5000回/日 ブロッカーの影響を受けない**。
予算課題は将来の Schedule／終了報告 と 見積/請求のライブ取得に局所化される。

## 6. AI / テンプレ

- 駆除作業報告の自由文は、施工内容テーブル＋定型文から生成（まずテンプレで足り、将来AI）。
- 報告内容は調査報告書・見積書から導出可能（やるべき作業＝報告項目）。本格活用は将来段階。
