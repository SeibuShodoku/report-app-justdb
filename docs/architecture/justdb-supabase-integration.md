# JUST.DB × Supabase 連携設計

> 2026-06 改訂。設計原則（収束・冪等・一方向／データプレーンとコントロールプレーンの分離／
> 契約は通常運転だけを縛る）を中核に据え、**API リソース予算（5000回/日）** を最優先制約として
> 明記した。現在、当日の API 消費の正体を別リポジトリで調査中（§4）。確定はそれを見てから。

## 1. 設計原則

この3つを外すと連携が「ヒッチャカメッチャカ」になる。手段（pull / CSV / Webhook）より先に、ここを固定点にする。

### 1-1. データプレーンとコントロールプレーンを分ける
- **データプレーン（バルク運搬）** = データフロー（CSV）＋単発APIリード/ライト。**Webhookで運ばない。**
- **コントロールプレーン（合図）** = Webhook。データを運ばず「終わった／失敗した／変わった」だけ伝える。
- Webhookは差分＝消費したら再読不能＝収束しない。枠の問題以前に、運搬には設計として不適。

### 1-2. 各経路は「収束・冪等・一方向」
- **収束** = 何回・いつ流しても、取りこぼしがあっても、正しい状態に着く。これが満たせれば監視も補正も要らない。
- 実現手段 = **再読込できる「状態」を引く（更新日時で絞ってよい）＋キーで upsert**。
- 全件置換は「更新日時を信じない時の特殊形（窓＝全部）」にすぎない。普段は更新日時の窓で十分軽い。
- イベント積み上げ（Webhook差分）は1個落とすと永久にズレる ＝ 採らない。

### 1-3. 契約が縛るのは「通常運転」。人の判断と例外は契約の外
- 契約 = 機械が毎日回す自動の流れ。ここを収束させる。
- 「削除する／なかったことにする」は人（やAI）が**自己責任で下す例外的判断**で、契約の外側。
- ヒューマンエラーまでデータモデルに吸わせると「契約のための契約」が肥大し、いずれ窒息する。
- 例外は **低頻度の全件照合（＝呼吸）** が静かに吸収する。これは命綱ではなく呼吸。

## 2. 役割分担

| 層 | 役割 |
|---|---|
| **JUST.DB** | 正本（システム・オブ・レコード）。事務データに特化。API上限 **5000回/日**（§4）。 |
| **Supabase** | マスタ（薬剤資材・顧客）のミラー。**ケースは持たない**（§3 Lane B でライブ取得）。 |
| **WEBアプリ** | 作成・AI/テンプレ・出力（PDF）。読み書きはサーバーのAPIルート経由（サービスロールキーは非公開）。 |
| **Google Drive** | 成果物の保管（PDF＋再編集JSON＋写真）。 |

## 3. 3レーン

方向と鮮度要求でレーンを分け、それぞれ「その実行だけで正しい状態に着地して終わる」ようにする。

### Lane A — マスタ（薬剤資材・顧客）→ Supabaseミラー
- 鮮度: 低（滅多に変わらない）。方式: **更新日時で増分pull**（§5）。
- 頻度: 日次（Cron）＋手動「同期」ボタン。範囲: 変更分のみ／たまに全件照合。
- Webhook: 0枠。APIコスト: 1〜数回/日（誤差）。

### Lane B — ケース（施工日程・案件・受注・見積）→ ライブ取得
- 鮮度: 高（その案件の最新が要る）。方式: **起動時に施工予定IDで1件だけ API GET**。
- スナップショットを持たない＝陳腐化しない。Webhook: 0枠。
- 帰結: **Supabaseはマスタ専用でよい**（ケースはミラーしない。現状のシード `construction_schedules` はスタブ）。

### Lane C — 書き戻し（金額・回数・薬剤・要約）→ ランディング1テーブル
- アプリは**フラット1行**を JUST.DB の**ランディング（総合）テーブルに POST**（1コール）。
- JUST.DB側の**データフロー**が分解して各テーブルに焼く。アプリは1スキーマにしか結合しない（内部スキーマ変更に強い）。
- 薬剤は名前でなく `justdb_id` 参照で渡す（ミラー済みのIDで分解側が確実に解決できる）。
- 二重送信は施工予定IDで upsert ＝ 冪等。完了/失敗の確証が要るなら Webhook 1枠を「分解完了通知」に使う（§6）。

## 4. API リソース予算（5000回/日）— 最優先の現場制約

設計の主戦場はここ。**薬剤同期ではなく「1報告あたりのコール数 × 報告数」が予算を食う。**

| 蛇口 | 方式 | 試算コール/日 |
|---|---|---|
| 薬剤・顧客マスタ同期 | 更新日時で増分pull | 1〜数回（全件照合でも10前後） |
| ケース取得 | 報告ごとに GET | 報告数 × 読みコール数 |
| 書き戻し | ランディングに POST | 報告数 × 1 |

- 増分pullは変更ゼロなら実質1コール。**マスタ同期は予算を脅かさない。**
- 予算最適化の的は **(a) ケース読みを少コールに、(b) 書き戻しをランディング1コールに** 抑えること。
- 例: 100報告/日 × (読み2＋書き1) ＋ マスタ5 ≒ **305回/日**（5000の約6%）。

### 現状（2026-06-17 22:00）と調査中の課題 ← 設計確定はこれ待ち
- **当日 4345/5000 を消費（残り約655）。正体不明の消費が約9割を占める。別リポジトリで内訳を調査中。**
- headroom が全レーンの可否を決めるため、判明するまで実装方式を確定しない。
- 持ち帰る観点: ①単一消費者か合算か ②ポーリングか活動比例か ③生きているか惰性か（未使用連携が焼いている可能性＝最有力で最も削れる）④読み/書き・枠は組織単位かキー単位か ⑤5000は引き上げ可能なプランか。
- 判明後の分岐:
  - **削れる** → Lane A は **pull増分のまま**で余裕。報告traffic も載る。
  - **削れず必須** → マスタを **データフローCSV（API 0）に逃がし**、報告あたりコールを最小化。本質（収束・冪等）はどちらでも不変。
- 調査タスクは `docs/spec/open-issues.md` §0。

## 5. 同期メカニズム（Lane A）

- **更新日時ウォーターマークでの増分upsert**。透かし `last_synced_at` を Supabase の sync_state に1行で保持。
- upsert キー = `justdb_id`。再実行しても二重化しない（冪等）。
- **削除の扱い**: 廃番は **論理削除（状態＝有効/廃番）を推奨運用** に。状態変更も更新日時を上げるので増分で拾える。
  - 物理削除は禁止しない（人の責任）。漏れは **低頻度の全件照合（呼吸）** が吸収する。
- カスケードは **状態＝有効** の薬剤のみ提示する。
- 実装手段（pull / メールCSV）の最終選択は §4 の調査結果で決める。本質（収束・冪等）はどちらでも不変。
- 現状: JUST.DB API 未接続。`docs/supabase/seed.sql` のシードが「同期済みの状態」を代替する（プレゼン用）。

## 6. Webhook予算（残り3枠）

| 枠 | 用途 | 種別 |
|---|---|---|
| ① | 書き戻し分解の完了/失敗をアプリへ通知（保存成功の確証） | 確認 |
| ② | 予備（将来のリアルタイム。例: 施工変更→Slack通知） | イベント |
| ③ | 安全マージン（残す） | — |

マスタ同期もケース取得も **0枠**。Webhookでデータを運ばない（§1-1）。

## 7. 起動とパラメータ（JUST.DBから）

JUST.DB側はレコードにアプリ起動リンクを保持し、起動時にキーをURLパラメータで渡す。

```
/report/new?caseId={案件ID}&investigationId={調査予定ID}&constructionId={施工予定ID}&driveFolderUrl={GoogleDriveフォルダURL}
```

- `caseId`（必須）/ `investigationId`（任意）/ `constructionId`（任意・=施工予定ID）
- `driveFolderUrl`（正規）/ `driveFolderId`（代替）/ `token`（任意・将来の署名検証）
- キー項目は原則編集不可。トークンは指定時のみ署名検証（当面トークンなし運用）。

（旧 JUST.DB連携仕様〔URLパラメータ方式〕を統合。旧版は `archive/integration-justdb-urlparam.md`。）

## 8. アンカーと結合チェーン

報告書 ↔ 施工 は **施工予定ID** で 1:1（施工日程テーブル）。

```
施工予定ID ──(FK)── 案件ID ── 受注ID / 見積書 / その他
```

起動時は施工予定ID（launch context の `constructionId`）を起点に、ケースデータ（顧客名・施工先・施工日時・報告日）を引く（Lane B）。

## 9. カスケード（施工内容の入力）

```
対象害虫 ──▶ 使用薬剤（適用害虫で絞り込み）──▶ 処理方法（薬剤ごと）
```

薬剤資材ミラー `chemicals` の各行が `applicable_pests`（適用害虫）と `methods`（処理方法）を持つ。
これだけで害虫→薬剤→処理方法が絞り込める。（実テーブルにこの関連が無ければ、Supabase側にこのミラー構造で持たせる。）

## 10. データ取得API（現状の実装）

| ルート | 用途 |
|---|---|
| `GET /api/master/pests` | 害虫マスタ（カスケード1段目） |
| `GET /api/master/chemicals?pest=ネズミ` | 適用薬剤＋各薬剤の処理方法 |
| `GET /api/case?constructionId=CONST001` | 施工予定IDでケースデータ（現状はSupabaseシード＝スタブ。目標はLane Bのライブ取得） |

すべて `src/lib/supabase-rest.ts` 経由で PostgREST に問い合わせる（SDK非依存）。

## 11. セットアップ

1. Supabase プロジェクトを作成。
2. SQL Editor で `docs/supabase/report-app-schema.sql`（テーブル）→ `seed.sql`（シード）を実行。適用順は `docs/supabase/README.md`。
3. `.env.local` に設定（**`.env.local` は Git 管理外**）:
   ```
   SUPABASE_URL=https://xxxx.supabase.co
   SUPABASE_SERVICE_ROLE_KEY=（Project Settings → API → service_role）
   ```
4. `npm run dev` → http://localhost:3000/mock
